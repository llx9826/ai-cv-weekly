# 任务目标

请基于当前代码库，对 LunaClaw Brief 报告系统做一次**最终整改**。

这次整改的目标不是做表面美化，也不是推翻重写，而是：

- 在保留现有 Python + Jinja2 + WeasyPrint 主架构的前提下
- 修复当前报告成品中的结构性问题
- 建立清晰的“内容层 / 结构层 / 展示层”边界
- 支持多主题渲染
- 提升报告的可读性、判断力、产品感
- 让最终输出不再像 debug 页面，而像真正的日报 / 周报产品

---

# 总体原则

## 1. 不要全面重构为 React
当前系统的主消费场景是：
- CLI / Skill
- 邮件投递
- PDF 导出

因此这次整改保持：
- Python pipeline
- Jinja2 模板
- HTML → PDF

但要在现有架构内完成：
- 模板分层
- 主题系统
- 结构化中间层
- 输出清洗
- 质量约束

---

## 2. 报告必须区分三层

### 数据层（内部）
原始 items、grounding、citation、eval、trace、stats 等内部数据。

### 结构层（中间）
统一的 ReportSchema，用于承载：
- hero_summary
- core_conclusions
- sections
- strategy
- risks
- appendix_notes

### 展示层（对外）
最终 HTML / PDF 成品，只允许输出给读者看的内容。

禁止将数据层和评估层内容直接暴露到展示层。

---

# 已确认需要修复的关键问题

请逐项落实，不要只做部分修补。

## 1. 修复 JSON / dict / 内部指标泄漏
当前 `stats` 中混入 dict 和内部指标，模板又无差别遍历，导致：
- dict 被直接渲染为字符串
- score / grounding / gate_verdict 等出现在最终报告

### 要求
- 将 `stats` 拆分为：
  - `reader_stats`
  - `internal_stats`
- 模板只能渲染 `reader_stats`
- `internal_stats` 不得进入最终模板上下文

### reader_stats 只允许包含面向读者的标量信息，例如：
- 信息源数量
- 选取素材数量
- 报告字数

禁止 reader_stats 中出现：
- dict
- score
- grounding
- verdict
- citations summary object

---

## 2. 重构 citation 展示方式
当前 citation appendix 展示所有 claim → evidence 映射，导致：
- 溯源表格过长
- 正文被附录污染
- 报告像审计日志
- 多条低置信度映射破坏信任

### 要求
- citation 对读者视图改为“摘要式来源说明”
- 不再展示大表格
- 仅展示高置信度来源摘要
- 最多保留 3~5 条来源
- 允许展示：
  - “本报告数据基于 GitHub、官方公告、公开项目页等来源验证”
- 禁止展示：
  - 逐条 claim 对照表
  - 低置信度 claim
  - “未溯源 0%”
  - “可信度 70%”

### 代码要求
- 在 renderer 层过滤 citation：
  - `confidence >= 0.7`
  - `top_k <= 5`
- 传给模板的不是 citation table，而是：
  - `citation_sources`
  - `citation_note`

---

## 3. 重构第一页顶部信息区（不是单独封面页）
注意：这里不是新增一页封面，而是重构第一页顶部的入口区域。

### 目标
用户在 3 秒内理解：
1. 这是什么报告
2. 本期最重要变化是什么
3. 时间范围是什么
4. 最值得先看的核心摘要是什么

### 顶部信息区必须包含
- 主标题
- 一句话核心判断（hero_summary）
- 时间范围
- 3~5 条核心摘要 bullet

### 布局要求
- 主标题：居中
- hero_summary：居中，视觉权重最高
- 时间：居中
- 核心摘要：左对齐 bullet
- 不得做成单独封面页
- 顶部信息区之后直接进入正文

### 说明
当前标题区即使已经 `text-align: center`，也不等于已经具备“信息入口”。  
这次整改要解决的是信息架构问题，而不是只做样式居中。

---

## 4. 统一所有 editor 的 hero + conclusion + strategy 规范
不同 editor 当前输出结构不统一，必须收敛。

### 所有 editor 都必须新增 / 强化以下输出要求

#### A. 顶部核心判断
在报告最前面必须输出一个“核心判断”区域，用一句话概括本期最重要变化。
要求：
- 不超过 30 字
- 是判断，不是描述
- 有信息增量
- 不允许空话

#### B. 核心结论
每条核心结论必须包含三要素：
- 结论（判断性陈述）
- 依据（数据 / 事实 / 事件）
- 约束（什么时候不成立）

#### C. 策略层
所有日报 / 周报类 editor 都必须有策略层。

策略层不等于投资建议，而是“行动纪律 / 投入判断 / 观察建议”。

必须包含：
- 当前阶段（进攻 / 防守 / 观察；对 tech 可用“投入期 / 跟踪期 / 观望期”）
- 不值得投入的方向
- 值得跟踪的方向
- 触发条件 / 前提条件
- 当前最大风险

禁止输出：
- 买入
- 卖出
- 加仓
- 个股推荐
- 收益暗示

## 5. 强化观点质量，抑制“正确废话”
当前 prompt 虽有限制，但还不够，必须在 quality 层增加硬约束。

### 在 quality.py 中新增空话检测
黑名单示例：

- 有待观察
- 具有重要意义
- 推动行业发展
- 持续关注
- 有望
- 或将
- 进一步推动
- 不断深化

### 规则
- 允许极少量出现
- 超过阈值则判定质量问题
- 在 issues 中明确提示“模板化表达过多”

### 额外要求
除黑名单外，还需检查：
- 核心结论是否具备“依据 + 约束”
- 策略层是否存在
- hero_summary 是否存在

### sanitizer 需要清洗的内容包括但不限于

- ```json code block

### 要求

### 在 models.py 中新增结构化 schema

### 推荐字段

#### Conclusion

### 实施方式

### 目标

- 支持不同模板与主题共享同一结构

## 8. 增加数值自洽校验
当前有 grounding，但缺少报告内部一致性检查。

### 新增 ConsistencyGrounder
至少检查：

- 百分比是否异常
- 基础汇总是否冲突
- 常见数值格式是否前后不一致

### 要求
- 注册到 grounding pipeline
- 发现矛盾时：
  - 优先记录 issue
  - 高风险时阻止输出或降级结论
