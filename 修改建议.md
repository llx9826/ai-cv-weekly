任务目标

将当前的日报 / 周报生成系统，重构为一个：

👉 “结构化内容 + 多主题前端渲染 + PDF 导出”的报告产品系统

目标不是修模板，而是构建一个：

- 可扩展
- 可切换主题
- 可产品化展示
- 可写入简历

的报告引擎。

---

一、核心问题（必须修复）

当前系统存在以下结构性问题：

1. JSON / dict / score / grounding 等中间态泄漏到最终报告
2. 溯源表格（30+条）污染正文，像 debug 日志
3. 第一屏没有信息入口，读者无法快速理解重点
4. 标题区没有视觉中心（未居中、无层级）
5. 核心结论被埋在正文，没有上提
6. 缺少“策略层 / 行动层”
7. 观点偏弱，存在“正确废话”
8. 数字一致性未校验（分项 ≠ 总和）
9. 模板与展示耦合严重，不支持多主题
10. 附录信息量过大，抢占正文视觉权重

---

二、技术架构（必须采用）

必须采用前后分离设计：

内容层（Content Layer）

由 LLM / pipeline 生成统一结构化数据（JSON）。

展示层（Presentation Layer）

使用 React + Tailwind 渲染报告页面。

主题层（Theme Layer）

控制视觉与信息密度（不可改内容逻辑）。

---

三、技术栈要求

- React + TypeScript
- Tailwind CSS
- shadcn/ui（可选）
- 组件化设计
- 支持浏览器导出 PDF

禁止：

- 继续用单一 Jinja2 模板承担所有展示逻辑

---

四、统一数据结构（必须设计）

必须定义统一 report schema，支持日报 / 周报 / AI / A股 等复用：

包含：

- meta
  - title
  - hero_summary（核心一句话）
  - date_range
  - report_type
- core_conclusions[]
  - conclusion
  - basis
  - constraint
- key_events[]
  - title
  - summary
  - logic
  - impact
  - claw_comment
- strategy
  - stage（进攻 / 防守 / 观察）
  - avoid
  - watch
  - trigger_conditions
  - biggest_risk
- risks[]
- appendix[]

---

五、第一页顶部信息区（重点重构）

⚠️ 注意：不是封面页，不单独占页

必须在第一页顶部完成信息入口设计。

---

目标

用户在 3 秒内理解：

1. 这是什么报告
2. 本期最重要变化
3. 时间范围
4. 核心摘要

---

布局（强制）

居中轴结构：

主标题（居中）
一句话核心判断（居中，最大）
时间范围（居中）

分隔线

核心结论（3~5条，左对齐 bullet）

---

示例

AI / CV Weekly Brief

OCR 正从模型竞争转向工程生态垄断

2026.03.10 – 2026.03.17

---

核心结论

• 工具链两极固化，中间层消失
• 工程整合优先于端到端模型
• 基础设施正在锁定创新路径

---

禁止内容（必须严格执行）

- JSON / dict
- score / grounding / citation
- total items / sources
- debug字段
- 溯源表格

---

六、正文结构（组件化）

必须拆为以下 React 组件：

- HeroHeader（顶部信息区）
- CoreConclusionsSection
- KeyEventsSection
- StrategySection
- RiskSection
- AppendixSection

---

1. CoreConclusionsSection

每条必须包含：

- 结论（判断）
- 依据
- 约束条件（什么时候不成立）

---

2. KeyEventsSection

每个事件包含：

- 事件
- 背后逻辑
- 行业影响
- CLAW锐评

---

CLAW锐评要求

必须：

- 有立场
- 有矛盾（表象 vs 本质）
- 有条件
- 不说废话

---

3. StrategySection（必须存在）

必须输出：

- 当前阶段（进攻 / 防守 / 观察）
- 哪类方向不值得投入
- 哪类方向可以跟踪（+前提条件）
- 当前最大风险

禁止：

- 买入 / 卖出 / 加仓
- 个股推荐
- 收益暗示

---

4. RiskSection

最多3条：

- 情绪风险
- 技术风险
- 基本面风险

必须具体。

---

5. AppendixSection

规则：

- 最多5条
- 不做表格
- 不逐条溯源
- 不出现“70% / 未溯源”

---

七、溯源系统（必须重构）

原则：

👉 服务阅读，而不是展示系统能力

禁止：

- 30+条溯源列表
- 句子拆碎
- source错位

允许：

- 摘要级来源说明（如：基于 GitHub 数据）

---

八、数学校验

必须实现校验逻辑：

- 分项是否等于总和
- 数字是否一致
- 单位是否统一

不一致：

- 修正或删除该结论

---

九、主题系统（必须实现）

同一份内容 → 多主题渲染

---

1. ClawCat Brief（默认）

- 极简
- 强结构
- 无干扰
- 决策导向

---

2. ClawCat Light

- 浅色
- 高可读性
- 长时间阅读友好

---

3. ClawCat Dense

- 高信息密度
- 紧凑布局
- 适合扫描

---

4. ClawCat Terminal

- 深色
- 内部调试风格
- 可展示更多技术信息

---

十、视觉规范（重点实现 Brief + Light）

Brief

- max-width: 720px
- 标题区居中
- 正文左对齐
- 黑白灰为主 + 少量强调色
- 强结构，低装饰

Light

- 浅色背景
- 更大行距
- 更舒适阅读

---

十一、工程输出要求

请直接输出：

1. React 项目结构
2. 组件拆分
3. theme token 设计
4. mock 数据
5. Brief + Light 主题实现
6. 主题切换方式
7. PDF 导出建议

---

十二、失败判定（任意触发即失败）

- 出现 JSON / dict
- 出现 score / grounding
- 出现溯源大表
- 第一页顶部信息区缺失或混乱
- 没有策略层
- 观点空洞
- 报告像 debug 输出

---

十三、最终目标

这个系统必须达到：

- 同一内容，多主题渲染
- 用户 3 秒看懂重点
- 报告不再像工具输出
- 可以作为产品展示
- 可集成到 openclaw skill
